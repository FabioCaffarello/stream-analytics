# W5.1 Sweep Throttling Hardening

Date: 2026-02-12

## Problema

`Sweep()` estava no hot path de get-or-create:
- `internal/core/marketdata/app/ingest.go`
- `internal/core/aggregation/app/update_orderbook.go`

Isso introduzia risco de custo `O(n)` por mensagem sob cardinalidade alta.

## Objetivo

Amortizar sweep por cadência (ops/tempo), preservando:
- semântica de TTL: `Get()` nunca devolve item expirado;
- semântica de LRU/maxSize;
- determinismo via `clock.Clock` (sem fontes novas de tempo global no core).

## Patch

### 1) Throttling no `BoundedMap`

Arquivo: `internal/shared/ds/boundedmap.go`

Mudanças principais:
- Novos campos internos:
  - `opCount uint64`
  - `sweepEveryOps uint64`
  - `sweepMinInterval time.Duration`
  - `lastSweep time.Time`
- Novos métodos opcionais (non-breaking):
  - `SetSweepEveryOps(n uint64)`
  - `SetSweepMinInterval(d time.Duration)`
- Novo fluxo:
  - `Get/Put/Delete` incrementam `opCount` e chamam `maybeSweepLocked(now)`.
  - `Sweep()` manual agora usa `sweepLocked(now)` interno.
- `OnEvict` permanece executando fora do lock.
- Defaults:
  - throttling desligado (`0`) por padrão.

### 2) Wiring nos hot paths

- `internal/core/marketdata/app/ingest.go`
  - `SetSweepEveryOps(1024)`
  - `SetSweepMinInterval(1 * time.Second)`
  - remoção de `uc.streams.Sweep()` por requisição.

- `internal/core/aggregation/app/update_orderbook.go`
  - `SetSweepEveryOps(1024)`
  - `SetSweepMinInterval(1 * time.Second)`
  - remoção de `uc.books.Sweep()` por requisição.

## Correção de semântica

TTL correctness continua garantida sem sweep imediato:
- `Get()` valida `expiresAt` e retorna miss para expirado.
- `Sweep` apenas limpa itens expirados não acessados (housekeeping), sem mudar contrato de leitura.

LRU/maxSize continua:
- Em `Put()`, quando `len >= maxSize`, evicção LRU por tamanho continua ativa.

## Evidências executadas

### Testes

Comando:
```bash
go test -race ./internal/shared/ds ./internal/core/marketdata/app ./internal/core/aggregation/app
```

Resultado: todos verdes.

### Benchmark comparativo

Comando:
```bash
go test ./internal/shared/ds -run '^$' -bench 'BenchmarkBoundedMapPutGet_(WithSweepEveryOp|ThrottledSweep)$' -benchmem -benchtime=20s -cpuprofile .context/evidence/w5.1-sweep-cadence-cpu.pprof -mutexprofile .context/evidence/w5.1-sweep-cadence-mutex.pprof -mutexprofilefraction=1
```

Saída:
- `BenchmarkBoundedMapPutGet_WithSweepEveryOp-10`: `5533 ns/op`, `135 B/op`, `4 allocs/op`
- `BenchmarkBoundedMapPutGet_ThrottledSweep-10`: `158.6 ns/op`, `138 B/op`, `4 allocs/op`

Interpretação: varredura amortizada elimina o custo de varredura completa por operação, reduzindo fortemente latência/op no cenário de expiração agressiva.
